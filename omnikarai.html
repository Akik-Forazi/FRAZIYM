Language & Compiler · v6.0 · Active

Omnikarai

A hand-written programming language with a compiler that emits native Windows x64 machine code directly — no LLVM, no GCC backend, no intermediate representation. Every byte of the output binary is authored by the compiler.

Built in C from scratch. Full Pratt parser. INDENT/DEDENT lexer. Direct x86-64 codegen. v6.0 ships with 20/20 benchmark tests passing — Fib(40) at 3.72x of GCC -O3, Primes at 3.15x of GCC -O3.

The Project

A compiler built the hard way

Most compilers stand on giants — LLVM handles register allocation, GCC handles instruction selection, and the language author handles syntax. Omnikarai removes every layer of that abstraction. The lexer, parser, semantic analysis, code generator, and binary emitter are all written by hand in C. The output is a raw Windows PE executable containing x86-64 machine code.

No Backend

Zero Dependencies

No LLVM. No GCC. No libgcc. The compiler links only against kernel32.dll — Windows' lowest-level system library. Every instruction in the output binary was decided by the codegen module.

Native Speed

Within 3–4x of GCC -O3

In the v6.0 benchmark suite, Omnikarai runs Fib(40) at 6683 ms vs C -O3's 1797 ms (3.72x ratio) and Primes at 1246 ms vs C's 396 ms (3.15x). Dotprod and Matmul are next targets for the register allocator phase.

Python-like Syntax

Familiar by Design

Omnikarai uses significant whitespace, colon-terminated blocks, and a clean keyword set. It feels like Python — but compiles to a native binary that needs no runtime, no interpreter, and no VM.

Language Features · v6.0

What the language supports today

Omnikarai v6.0 is a fully functional language covering the core of structured programming. Every feature listed below compiles to native x86-64 code and has been verified through the 20-test benchmark suite.

🔨

Full Pratt Parser

Recursive descent parser with Pratt operator precedence. Handles nested expressions, chained comparisons, logical operators (and, or, not), and the full arithmetic suite including modulo.

📝

INDENT/DEDENT Lexer

Python-style significant whitespace. The lexer emits synthetic INDENT and DEDENT tokens so the parser never sees raw spaces — enabling clean block-structured grammar without braces.

⚙

Functions & Recursion

User-defined functions with arguments, return values, and full recursion support. Nested function calls, mutual recursion, and deeply nested call stacks — all generating correct stack frames.

🔄

Control Flow

if / elif / else with arbitrary nesting and blank lines between blocks. while loops with break and continue. for i in range(n) iterators. All emit correct x86-64 conditional jump sequences.

🎯

match / case

Structural pattern matching with wildcard _ case. Emits a comparison chain with direct jump dispatch. Cleaner than chained if/elif for switch-style logic.

📚

Standard Library

Built-in print(), str(), int(), len(), and input() — all implemented as native function calls that interface directly with Win32 console I/O. No libc dependency.

Compiler Architecture

From source to binary — four stages

The compiler pipeline is four hand-written C modules with no shared-library or third-party dependencies. Each stage is independently testable and produces a well-defined output that the next stage consumes.

STAGE 01 — lexer.c

Lexer

Converts source text into a flat token stream. Tracks indentation levels to synthesise INDENT and DEDENT tokens. Handles string literals, numeric literals, keywords, operators, and identifiers. Output: Token[]

STAGE 02 — parser.c

Pratt Parser

Top-down Pratt parser producing an Abstract Syntax Tree. Prefix and infix parse functions registered per token type. Handles operator precedence, block statements, and all compound constructs. Output: AST_Program*

STAGE 03 — codegen.c

x86-64 Code Generator

Walks the AST and emits raw x86-64 machine code bytes. Manages stack frames, variable offsets via rbp-relative addressing, conditional jumps with backpatching, and function call conventions (System V on Windows). Output: byte[]

STAGE 04 — main.c

PE Binary Emitter & Runner

Wraps generated machine code in a Windows PE header, allocates executable memory via VirtualAlloc, and either writes a .exe to disk or executes in-process for omnicc run mode. No intermediate object files.

Build Phases

Omnikarai development roadmap

Phase 1 — Core Language

Variables, arithmetic, comparisons, logical operators, if/elif/else with nesting and blank lines between blocks. Full INDENT/DEDENT lexer and Pratt parser.

Phase 2 — Control Flow

while loops, break, continue, for i in range(n). All emit correct conditional jump sequences with backpatching in the codegen layer.

Phase 3 — Functions & match/case

User-defined functions with arguments, return, recursion, and nested calls. match/case pattern matching with wildcard support. Standard library builtins: print, str, int, len, input.

Phase 4 — Runtime Type System Next

List literals [a,b,c], dict literals {"k":v}, index access obj[i], append(), string concatenation, and float arithmetic codegen. Introduces a tagged value representation.

Phase 5 — Register Allocator

Move loop induction variables and hot scalars from rbp-offset stack slots into CPU registers. Expected to close the remaining performance gap vs GCC -O2 on non-recursive workloads.

Phase 6 — First-Class Functions & Closures

Functions as values, closures over enclosing scope, and higher-order functions. Enables functional programming patterns natively.

Phase 7 — Modules & Standard Library

Multi-file compilation, import system, and an expanded standard library covering file I/O, string operations, and basic collections.

Context

Why build a compiler?

Omnikarai started as a question: what does it actually take to turn text into a running program? Not "what does LLVM do under the hood" but "what are the exact bytes emitted, and why?"

It sits alongside XORZEN and AXONIX-ZERO as a demonstration that Akik's engineering instinct operates at every level of the stack — from designing sparse attention mechanisms, to writing autonomous agent loops, to encoding x86-64 instruction bytes by hand.

The v6.0 benchmark suite — 20/20 tests passing — shows Fib(40) at 3.72× and Primes at 3.15× of GCC -O3. The larger gaps on Matmul and loop-heavy workloads are the direct target of Phase 5's register allocator, which will eliminate the rbp-offset memory round-trips currently limiting throughput.
